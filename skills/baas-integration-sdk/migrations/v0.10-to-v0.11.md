# v0.10 → v0.11 마이그레이션

**결제 세션 앵커 통일 + 가상계좌 웹훅.** 백엔드가 결제 축을 `PaymentSession`(CREATED→PAID)으로 통일하고,
결제 완료를 **카드=동기 confirm / 가상계좌=비동기 웹훅(PAYMENT_STATUS_CHANGED·DEPOSIT_CALLBACK)** 로 처리한다.
SDK는 결제 세션+도메인 레코드를 **위젯 진입(`beginWidgetCheckout`) 시점에 `start` 로 생성**하도록 내부 배선을
바꿨다(v0.10은 prepare만, 생성은 confirm에서 했음). **앱이 쓰는 handle 형태(`orderNo`/`orderId`)와 confirm
시그니처는 그대로라 대체로 비파괴적**이다(런타임 CDN이라 앱은 자동으로 최신 SDK를 로드).

## 무엇이 바뀌었나
- **동작(내부)**: `beginWidgetCheckout` 이 위젯 진입 시 `start`(store `/orders/checkout/start`, 예약
  `/bookings/start`)로 결제 세션(CREATED)+도메인 레코드(주문/예약 PENDING)를 만든다. order_no 를 이때 확보한다.
- **⚠ 핵심 규칙**: `handle.requestPayment` 는 **결제 버튼 클릭 핸들러 안에서 동기로** 호출해야 한다(앞에
  `await`/비동기 작업을 넣지 말 것). **현대카드 등 팝업·앱카드 결제창은 사용자 제스처가 끊기면 열리지 않는다.**
  그래서 order_no 를 클릭 전(위젯 진입 시)에 미리 만들어 둔다. (v0.11.0 에서 잠깐 requestPayment 앞에 start 를
  두어 현대카드가 안 뜨던 이슈 → v0.11.1에서 start 를 진입 시점으로 되돌려 수정.)
- **신규(선택)**: `useStore().getCheckoutContext()`/`clearCheckoutContext()` — 복귀 confirm용 컨텍스트
  `{ order_no, product_id, quantity }`(예약은 기존 `useReservation().getCheckoutContext()` 그대로).
- **가상계좌 지원**: 입금은 서버 웹훅이 완결하므로, 복귀 시점에 주문/예약이 아직 PENDING(입금 대기)일 수 있다.
- **불변**: `beginWidgetCheckout`/`handle.orderNo`·`handle.orderId`/`confirm({ order_no, payment_key, amount, ... })`
  는 그대로. order_no 는 handle 또는 successUrl 쿼리(orderId)로 얻어 confirm 에 넘긴다(v0.10과 동일).

## 영향 / Tier (판정: features_used ∩ {payment, store, reservation})
- **Tier 0**: 결제(store·reservation·결제) 미사용 앱 → 무영향(런타임 SDK만 최신).
- **Tier 1 (자동/무수정에 가까움)**: 위젯 결제 앱 대부분 — handle 형태·confirm 이 그대로라 코드 변경 없이 동작한다.
  단, **결제 버튼 핸들러에서 `requestPayment` 앞에 `await`(네트워크 호출 등)를 두지 않았는지**만 점검한다
  (두었다면 현대카드 등 팝업 결제가 깨지므로 제거 — sessionStorage 저장 같은 동기 작업은 무방).

## 적용 절차 (앱빌더/에이전트)
1. 런타임 SDK(next 채널)는 자동 최신. 앱 코드 점검만 한다.
2. **결제 버튼 클릭 핸들러**: `handle.requestPayment(...)` 직전에 `await` 로 비동기 작업을 넣지 않는다(있으면 제거).
   위젯 진입(`beginWidgetCheckout`)에서 이미 order_no 가 확보되므로 클릭 시엔 동기 호출만 하면 된다.
3. **복귀 confirm**: 기존대로 successUrl 쿼리(orderId/paymentKey/amount) 또는 `getCheckoutContext()` 로
   `confirm({ order_no, payment_key, amount, ... })`. (store 는 product_id/quantity, 예약은 reserved_at/form_data
   를 컨텍스트/쿼리로 함께 — 서버는 order_no 로 레코드를 로드하므로 값이 정확하기만 하면 된다.)
4. (선택) 가상계좌를 쓰면 복귀 화면에 "입금 확인 후 완료" 안내(입금 완료는 웹훅 처리).
5. `npm run validate && npm run build` 후 재배포. `baas-manifest.json` 의 `skill_version` → `"0.11.1"`.
