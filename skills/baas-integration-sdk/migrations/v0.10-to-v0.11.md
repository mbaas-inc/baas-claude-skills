# v0.10 → v0.11 마이그레이션

**결제 세션 앵커 통일 — 세션/도메인 레코드를 "결제하기" 클릭 시점에 생성.** 백엔드가 결제 축을
`PaymentSession`(CREATED→PAID)으로 통일하면서, SDK도 결제 세션·주문/예약을 위젯 열 때가 아니라 **결제 버튼
클릭(`handle.requestPayment`) 시점에 백엔드 `start` 로 생성**하도록 바뀌었다. 결제 완료는 **카드=동기 confirm /
가상계좌=비동기 웹훅**으로 처리된다. pre-1.0 정리라 하위호환 별칭 없이 handle 형태를 변경했다(BREAKING).

## 무엇이 바뀌었나
- **동작 변경**: `beginWidgetCheckout` 은 위젯 렌더용 금액만 확정하고(주문·세션 미생성 — 위젯만 열고 이탈해도
  잔재 없음), `handle.requestPayment` 호출 시점에 SDK가 `start`(store `/orders/checkout/start`, 예약
  `/bookings/start`)로 결제 세션+도메인 레코드(주문/예약 PENDING)를 만든 뒤 토스 결제를 요청한다.
- **제거(BREAKING)**:
  - `StoreWidgetHandle.orderNo` 제거 — order_no 는 결제하기 클릭 시 발급되므로 사전 노출 불가.
    복귀 페이지에서 `useStore().getCheckoutContext()` 또는 successUrl 쿼리(orderId)로 얻는다.
  - `ReservationWidgetHandle.orderId` 제거 — 동일 사유. `useReservation().getCheckoutContext()` 사용.
- **신규**:
  - `useStore().getCheckoutContext()` / `clearCheckoutContext()` — 복귀 confirm용 컨텍스트
    `{ order_no, product_id, quantity }`(sessionStorage, start 시점 저장). 예약은 기존과 동일하게
    `useReservation().getCheckoutContext()` → `{ target_id, order_no, reserved_at, form_data }`.
  - 코어 `BaasSDK.startOrder` / `BaasSDK.startBooking`(보통 직접 호출 불필요 — `beginWidgetCheckout` 내부에서 사용).
- **완료 처리 이원화**: successUrl 복귀 confirm 은 **카드(동기)** 완결 + 즉시 UX용. **가상계좌(비동기)** 는
  입금 시 서버 웹훅(PAYMENT_STATUS_CHANGED / DEPOSIT_CALLBACK)이 완결하므로, 복귀 시점엔 "입금 대기"일 수 있다.
- **confirm 필드 불변**: `confirm({ order_no, payment_key, amount, ... })` 시그니처는 그대로(서버가 order_no로
  레코드를 로드). 다만 값의 출처가 handle 사전값 → getCheckoutContext 로 바뀐다.

## 영향 / Tier (판정: features_used ∩ {payment, store, reservation})
- **Tier 0**: 결제(store·reservation·결제) 미사용 앱 → 교집합 없음 → 무영향(런타임 SDK만 최신).
- **Tier 2 (국소 수정)**: `beginWidgetCheckout` 로 위젯 결제를 구현한 앱 — handle 사전값(`w.orderNo`/`w.orderId`)을
  참조하거나 복귀 confirm 에 product_id/quantity·reserved_at/form_data 를 앱 상태/쿼리로 넘기던 앱.

## 적용 절차 (앱빌더/에이전트)
1. **위젯 렌더 코드**: `beginWidgetCheckout(...)` 호출부는 그대로. 다만 반환 handle 에서 `w.orderNo`/`w.orderId`
   를 읽던 코드를 제거한다(더 이상 없음 — 결제하기 클릭 시 발급).
2. **store 복귀 confirm**: successUrl 페이지에서
   `const ctx = useStore().getCheckoutContext();` →
   `await s.confirm({ order_no: ctx.order_no, payment_key, amount, product_id: ctx.product_id, quantity: ctx.quantity });`
   → `s.clearCheckoutContext();` (successUrl 에 product_id/quantity 쿼리를 싣던 방식은 불필요).
3. **예약 복귀 confirm**: 기존과 동일(`getCheckoutContext()` → confirm → clear). handle 에서 `w.orderId` 참조만 제거.
4. **가상계좌 UX**(선택): 복귀 시 주문/예약이 아직 PENDING(입금 대기)일 수 있으니 "입금 확인 후 완료" 안내를
   보이면 좋다(입금 완료는 웹훅이 처리). 카드만 쓰면 복귀 confirm 으로 즉시 완료.
5. `npm run validate && npm run build` 확인 후 재배포. `baas-manifest.json` 의 `skill_version` → `"0.11.0"`.

### 참고: 대부분 위젯 결제 화면 국소 수정
결제 실행 진입점(`beginWidgetCheckout`)·confirm 시그니처는 유지되므로, handle 사전값 제거 + 복귀 confirm 을
`getCheckoutContext` 기반으로 바꾸는 국소 수정이면 충분하다(Tier 2).
