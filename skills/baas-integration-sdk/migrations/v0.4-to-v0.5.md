# v0.4 → v0.5 마이그레이션

동적 컬렉션(`collection`) **SDK 사용 가이드 보강** 릴리스 — SDK 표면·서버 동작은 불변, 소비 방법의
문서 공백만 메움. (실 E2E에서 드러난 두 오류: `records` 봉투를 배열로 `.map`, union read를 관리 뷰로
그대로 렌더 → 403.)

## 무엇이 바뀌었나 (스킬 문서만)
- **`useCollection().records` 는 봉투** `{ items, total_count, offset, limit }` — 렌더는 항상
  `(records?.items ?? []).map(...)`. `records.map(...)` 은 `TypeError`.
- **union read 소비 규약**: read grants 가 `["owner","ref_owner:<f>"]` 면 `fetchRecords` 는 합집합을
  반환하고 **행별 매칭 자격을 표시하지 않는다**. 관리 뷰로 그대로 렌더 금지 —
  `r.account_id === user.id`(내가 낸 것) vs `!== user.id`(내가 부모 소유자로서 받은 것)로 나눠 소비,
  `ref_owner` 전용 액션(수락/거절 등)은 "받은 것" 부분집합에만 노출.
- SDK 코드/서버 access 모델/스키마 문법 변화 **없음**.

## 영향 / Tier (판정: features_used ∋ `collection`)
- **Tier 0 부분**: 단일 컬렉션(owner/public)만 쓰는 앱은 무영향 — union read 관리 뷰가 없으면 해당 없음.
- **Tier 2 부분(국소 수정)**: `ref_owner` 를 쓰는 **자식 컬렉션 관리 화면**(신청/댓글 관리)이 있는 앱만:
  - union 결과를 `account_id` 로 분리하도록 목록 컴포넌트 국소 수정.
  - `ref_owner` 전용 액션 버튼을 "받은 것" 부분집합에만 노출.
  - `records` 를 배열로 다루는 코드가 있으면 `records.items` 로 교정.

## 적용 절차 (앱빌더/에이전트)
1. `baas-manifest.json.features_used` 에 `collection` 없으면 건너뜀.
2. `ref_owner` access 를 쓰는 자식 컬렉션 + 그 관리 화면이 없으면 Tier 0 — manifest 버전만 갱신.
3. 있으면 관리 목록을 관점 분리(account_id)로 국소 수정 + `records.items` 렌더 확인 → 재빌드.
4. `baas-manifest.json.skill_version` 을 `0.5.0` 으로 갱신.

## 적용 후 확인
- [ ] 관리 화면에서 **내가 낸 신청 행에는 수락/거절 버튼이 없다**(받은 신청 행에만 있음).
- [ ] 받은 신청을 수락/거절하면 200(내 신청을 수락 시도 → 없어야 하지만, 있었다면 403).
- [ ] 목록 렌더가 `records.items` 기반이고 `records.map` 런타임 오류가 없다.
- [ ] 재빌드 성공 + `baas-manifest.json.skill_version` = `0.5.0`.

## codemod
없음 — 관점 분리는 화면별 판단이라 기계적 치환 불가(Tier 2 국소 수정). `records`→`records.items` 는
규칙적이나 오탐 위험(다른 `records` 식별자)으로 codemod 대신 수동 확인.
