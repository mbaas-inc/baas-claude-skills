# v0.3 → v0.4 마이그레이션

동적 컬렉션에 **관계 프리미티브**가 추가된 릴리스 (aiapp-service #626): reference 무결성 +
관계 파생 인가(`ref_owner`) + access grants 배열(OR).

## 무엇이 바뀌었나
- `settings.access` 값이 **atom 또는 배열(OR 합집합)** — 문자열(0.3)도 계속 유효(하위호환).
  새 atom `ref_owner:<field>` = 그 레코드의 reference 필드가 가리키는 **부모 레코드의 작성자**.
- **reference 무결성 서버 강제**: `submitRecord`/`editRecord`에서 reference 값이 대상 컬렉션의
  실존 레코드가 아니면 400. self-reference(같은 컬렉션) 허용.
- `read` grants 가 `["owner","ref_owner:<f>"]` 면 `fetchRecords`가 **내 레코드 + 내가 주인인
  부모에 달린 레코드**를 반환(관리 화면용 스코핑 — 클라 필터 불필요).
- 관계 설계 패턴 확정: 자식 컬렉션(신청·댓글)=reference+ref_owner / 트리=root anchor+parent
  이중 참조(anchor 평면 조회+클라 조립) / N:M=정션 컬렉션.
- SDK 표면 시그니처 **불변**(`useCollection` 훅 그대로) — 전송·인가는 SDK·서버가 흡수.

## 영향 / Tier (판정: features_used ∋ `collection` 인 앱만)
- **Tier 0 부분(자동)**: 기존 문자열 access·기존 화면은 그대로 동작(하위호환) — 앱 수정 불필요.
- **Tier 2 부분(국소, 해당 요구가 있을 때만)**: "부모 글에 달리는 데이터"(신청·댓글·참가)를
  0.3에서 우회 구현한 앱(예: read:member 로 열어두고 클라에서 필터)은 access 를 grants 배열로
  재설계하면 서버 강제로 전환 가능 — 컬렉션 관련 파일만 국소 수정:
  - 자식 컬렉션 access 를 `{"read":["owner","ref_owner:<ref필드>"],"update":["ref_owner:<ref필드>"],"delete":["owner","ref_owner:<ref필드>"]}` 로 재생성.
  - 관리 화면은 `fetchRecords` 결과를 그대로 렌더(클라 스코핑 코드 제거).
  - 부모 소유자 전용 버튼은 부모 레코드의 `account_id === user.id` 로 노출 제어(서버 강제의 보조).

## 적용 절차 (앱빌더/에이전트)
1. `baas-manifest.json.features_used` 에 `collection` 이 있으면 대상, 없으면 건너뜀.
2. 자식-컬렉션/트리 요구가 없으면 Tier 0 — manifest 버전만 갱신.
3. 있으면 해당 컬렉션을 grants 배열로 재생성(CLI `--access-json`) + 관련 화면 국소 수정.
4. 재빌드 → 재배포. `baas-manifest.json.skill_version` 을 `0.4.0` 으로 갱신.

## 적용 후 확인
- [ ] reference 필드에 임의 uuid 를 넣은 제출이 400 으로 거부된다(무결성).
- [ ] 부모 글 주인 계정으로 자식 목록(`fetchRecords`)에 타인 레코드가 보인다(ref_owner read).
- [ ] 부모 글 주인이 아닌 회원의 수락/거절(update)·모더레이션(delete)이 403 이다.
- [ ] 작성자 본인의 취소(delete:owner)는 동작한다.
- [ ] 재빌드 성공 + `baas-manifest.json.skill_version`이 `0.4.0`으로 갱신됐다.
