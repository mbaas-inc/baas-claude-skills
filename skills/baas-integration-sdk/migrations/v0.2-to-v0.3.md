# v0.2 → v0.3 마이그레이션

동적 컬렉션(`collection`) 접근 인가가 **CRUD 연산별 access 맵**으로 재설계된 릴리스.

## 무엇이 바뀌었나
- 컬렉션 접근 정책이 연산별 맵으로 통합: `settings.access = { create, read, update, delete }`,
  각 scope ∈ `public | member | owner`.
  - 기본값: `create:member`, `read:member`, `update:owner`, `delete:owner`.
  - 옛 불리언(`require_login`·`owner_scope`·`public_read`) 폐기.
  - 서버가 단일 choke point에서 강제(클라이언트 게이팅은 보조).
- CLI: `baas collection create` 정책 플래그가 `--access-json`(달라지는 연산만 부분 덮어쓰기)로 교체.
- SDK 표면(`fetchRecords`/`fetchPublicRecords`/`submitRecord`/`editRecord`/`removeRecord`,
  `useCollection`)의 **시그니처는 불변** — 전송·인가는 SDK·서버가 흡수(런타임 자동).

## 영향 / Tier (판정: features_used ∋ `collection` 인 앱만)
- **Tier 0 부분(자동)**: 전송·서버 인가 변경은 SDK가 흡수 → 배포 앱은 재생성 없이 동작 유지.
- **Tier 2 부분(국소 재생성)**: 0.2 이전 생성 앱의 **컬렉션 화면**이 접근 스코프에 맞는 UI 게이팅을
  반영하도록, 교집합 파일(컬렉션 목록/폼/상세 페이지·`useCollection` 사용부)만 국소 수정 권장:
  - `read:public` → 비로그인 목록/상세 열람 허용(`fetchPublicRecords` 경로).
  - `create:member` → 등록 액션은 `RequireAuth` 뒤.
  - `update/delete:owner` → 수정/삭제 버튼을 작성자 본인(`record.account_id === user.id`)에게만 노출.
- 순수 시각(색·여백·문구·레이아웃) 변경은 없음.

## 적용 절차 (앱빌더/에이전트)
1. `baas-manifest.json.features_used` 에 `collection` 이 있으면 대상, 없으면 건너뜀.
2. 컬렉션 관련 소스만 컨텍스트로 국소 재생성(전체 재생성 아님).
3. 재빌드 → 재배포. `baas-manifest.json.skill_version` 을 `0.3.0` 으로 갱신
   (`sdk_channel` 이 불변 버전이면 환경 채널(`dev=next`/`prod=v1`)로 함께 교정).

## 적용 후 확인
- [ ] `read:public` 컬렉션이면 **비로그인** 상태에서 목록/상세 열람이 동작한다(`fetchPublicRecords` 경로).
- [ ] `create:member`면 비로그인 시 등록 액션이 `RequireAuth`로 막힌다.
- [ ] `update/delete:owner`면 타인 레코드에 수정/삭제 버튼이 노출되지 않는다
      (`record.account_id === user.id` 조건 확인).
- [ ] 재빌드 성공 + `baas-manifest.json.skill_version`이 `0.3.0`으로 갱신됐다
      (`sdk_channel`이 불변 버전이었다면 환경 채널로 교정됐는지 포함).
